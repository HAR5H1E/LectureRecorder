## Refactoring and Unit Testing

*   **Refactoring**: Modify your code to make it more testable without affecting program correctness. This includes renaming variables, moving code around, and separating the Mastermind game logic from code that interacts with the human user via input/output (such as keyboard scanner classes).
*   **Unit Tests**: Use the `TestMastermindTest` class, which contains several unit test stubs, to ensure program logic works.
*   **Coverage Requirement**: Maintain over **90%** statement and branch coverage of your code using Eclipse.
*   **Test-Driven Robustness**: While Project 1 was graded based on a cooperative user, Project 2 will be graded by attempting to break your methods. You must add new test cases using JUnit assertions to check that invalid inputs result in exceptions being thrown.

## Exception Handling

To handle issues of data integrity at runtime, you must use exceptions to reject any guess or answer that violates game constraints. 
*   **Checked Exceptions**: Create two new checked exceptions that the `Mastermind` class must handle to enforce valid user input:
    *   `MastermindIllegalColorException` (raised if a method is provided with a guess or answer containing a color not part of the original six colors).
    *   Capital exception 2: Capital master Capital length capital exception (raised in case of an invalid guess or answer length).
*   **Error Handling**:
    *   If an exception is raised, the system must return a clear message indicating why the input was illegal.
    *   Do not count invalid guesses as one of the ten allowed guesses in the game.
    *   For any other issue, use standard Java class library exceptions (such as `IllegalArgumentException` or `IndexOutOfBoundsException`) to report it.
